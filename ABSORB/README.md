# 무작위 아이디어 단어 프로젝트 학습 교재

## 교재의 목적

이 문서는 완성된 화면만 소개하지 않는다. 원본 단어 파일이 브라우저용 데이터로 바뀌고, 사용자의 입력이 중복 없는 무작위 결과로 이어지는 과정을 직접 설명하고 수정할 수 있게 하는 것이 목적이다.

학습을 마치면 다음을 할 수 있어야 한다.

- 프로젝트의 목적과 GitHub Pages를 선택한 이유를 설명한다.
- HTML, CSS, JavaScript, JSON 파일의 역할과 연결 관계를 설명한다.
- 원본 문장 TSV에서 웹용 JSON을 다시 생성한다.
- 로딩, 입력 검증, 추첨의 실행 흐름을 추적한다.
- 작은 UI 또는 추첨 규칙 변경 지점을 찾는다.
- 데이터 로딩 실패를 순서대로 진단한다.

## 학습 대상과 선수 지식

HTML 태그, CSS 선택자, JavaScript 함수와 배열의 기초를 아는 학습자를 대상으로 한다. Python의 파일 읽기와 반복문을 본 적이 있으면 데이터 생성 스크립트를 이해하기 쉽지만 필수는 아니다.

## 프로젝트 소개

이 프로젝트는 한국어 뉴스 문장을 Okt로 분석해 명사·동사·형용사 기본형만 추출하고, 그 실질 형태소를 균등 확률로 뽑는다. 결과 단어들을 서로 연결해 새로운 제품, 이야기, 연구 주제 같은 아이디어를 떠올리는 데 사용한다.

## 프로젝트 지도

```text
원본 문장 TSV
  └─ scripts/build_words.py
       └─ data/words.json
            └─ app.js
                 ├─ index.html의 결과와 상태 갱신
                 └─ styles.css의 중앙 흑백 화면
```

주요 파일:

- [`../index.html`](../index.html): 결과, 입력 필드, 버튼, 상태 메시지의 구조
- [`../styles.css`](../styles.css): 중앙 정렬, 흑백 색상, 모바일 및 긴 단어 처리
- [`../app.js`](../app.js): JSON 로딩, 입력 검증, 중복 없는 추첨
- [`../scripts/build_words.py`](../scripts/build_words.py): 원본 문장 TSV를 JSON으로 변환
- [`../data/words.json`](../data/words.json): GitHub Pages가 직접 제공하는 단어 배열

## 전체 실행 흐름

1. 브라우저가 `index.html`을 읽고 CSS와 JavaScript를 불러온다.
2. 입력 필드와 버튼은 데이터가 준비될 때까지 비활성화되어 있다.
3. `loadWords()`가 `data/words.json`을 요청하고 배열인지 검사한다.
4. 성공하면 조작을 활성화하고, 실패하면 새로고침과 파일 확인 방법을 표시한다.
5. 사용자가 1~100의 정수를 입력하고 추첨을 실행한다.
6. `pickUniqueWords()`는 임의 인덱스를 `Set`에 모아 중복을 제거한다.
7. 선택된 단어를 줄바꿈으로 연결해 `<output>`에 표시한다.

## 핵심 개념

### 정적 배포와 사전 변환

GitHub Pages는 서버 코드를 실행하지 않는다. 따라서 큰 문장 TSV와 형태소 분석기를 브라우저에서 처리하지 않고, 배포 전에 명사·동사·형용사 47,203개를 약 0.5MB의 JSON 배열로 만든다. 원본은 보존되고 브라우저의 작업량은 줄어든다.

다시 생성하려면 저장소 루트에서 다음을 실행한다.

```powershell
python -m pip install -r requirements.txt
python scripts/build_words.py
```


### 형태소 분석과 기본형

[`../scripts/build_words.py`](../scripts/build_words.py)는 [`KoNLPy Okt`](https://konlpy.org/en/latest/api/konlpy.tag/)의 `pos()`를 사용한다. `norm=True`는 비표준 표현을 정규화하고, `stem=True`는 `했다`를 `하다`, `아름다운`을 `아름답다`처럼 기본형으로 바꾼다. 이 프로젝트는 `Noun`, `Verb`, `Adjective` 태그만 남기고 조사, 어미, 접속사, 문장부호는 제외한다.

형태소 분석에는 Python 3, Java 8 이상과 [`../requirements.txt`](../requirements.txt)의 KoNLPy가 필요하다. 이미 생성된 웹페이지를 사용하는 데에는 이 의존성이 필요하지 않다.

### 비동기 로딩

`fetch()`가 끝나기 전에는 단어 배열이 비어 있다. 이때 추첨하면 올바른 결과를 만들 수 없으므로 버튼과 입력을 잠그고 로딩 문구를 표시한다. 성공과 실패는 `try`/`catch`로 나누어 처리한다.

### 균등하고 중복 없는 추첨

빈도 열은 사용하지 않으므로 JSON 배열의 각 항목은 같은 확률을 가진다. `Set`은 이미 나온 인덱스를 다시 저장하지 않기 때문에 한 추첨 안에서 같은 항목이 반복되지 않는다. 최대 100개만 뽑으므로 47,203개 배열 전체를 섞는 것보다 필요한 인덱스만 고르는 방식이 단순하고 효율적이다.

## 직접 관찰하고 수정하기

1. 추첨 전에 결과와 로딩 상태가 어떻게 바뀌는지 예측한다.
2. 입력값을 `1`, `3`, `100`, `0`, `101`, 빈 값으로 바꾸어 차이를 관찰한다.
3. `styles.css`의 `font-size`만 바꾸고 좁은 화면에서 결과를 확인한다.
4. 원본을 보지 않고 `readCount()`와 `pickUniqueWords()`를 빈 파일에 다시 작성한다.
5. 왜 전체 배열을 매번 섞지 않았는지 자신의 말로 설명한다.

## 디버깅 순서

1. 브라우저 개발자 도구의 Network에서 `data/words.json`이 200 응답인지 본다.
2. 파일이 없다면 `python scripts/build_words.py`를 실행했는지 확인한다.
3. JSON 오류라면 생성 스크립트의 완료 메시지와 파일 시작 문자를 확인한다.
4. 입력 오류라면 값이 정수이며 1~100 범위인지 확인한다.
5. GitHub Pages에서만 실패하면 저장소 루트가 배포 원본인지, 파일명 대소문자가 정확한지 확인한다.

`index.html`을 더블클릭하면 브라우저의 로컬 파일 보안 정책 때문에 `fetch()`가 실패할 수 있다. 실제 조건과 비슷하게 확인하려면 저장소 루트에서 정적 HTTP 서버를 사용한다.

## 이해 확인 퀴즈

1. **기본·개념 설명형:** 원본 문장 TSV를 브라우저에서 바로 읽지 않고 JSON으로 변환한 이유를 두 가지 설명하라.
2. **적용·코드 추적형:** 페이지를 연 직후부터 버튼이 활성화될 때까지 호출되는 함수와 화면 변화를 순서대로 설명하라.
3. **변형·오류 찾기형:** `data/words.json`의 경로를 잘못 바꾸면 사용자는 무엇을 보며, 개발자는 무엇부터 확인해야 하는가?
4. **적용·결과 예측형:** 이미 선택된 인덱스가 다시 생성되면 `Set`의 크기와 반복문은 어떻게 되는가?
5. **독립 수행·구현형:** 1~50만 허용하도록 바꾸려면 HTML과 JavaScript에서 각각 무엇을 수정해야 하는가?
6. **독립 수행·구두 설명형:** 빈도 가중 추첨과 현재 균등 추첨의 차이, 그리고 현재 방식을 선택한 이유를 설명하라.
7. **적용·코드 추적형:** `있는`, `했다`, `아름다운`, `정책을`을 Okt가 분석할 때 저장되는 형태소와 제외되는 요소를 예측하라.

정답을 보기 전에 코드 위치와 실행 흐름을 근거로 답을 작성한다. 해설은 [`solutions/README.md`](solutions/README.md)에 있다.

## 최종 점검

- 목적과 전체 데이터 흐름을 파일 위치와 함께 설명할 수 있는가?
- `loadWords()`, `readCount()`, `pickUniqueWords()`를 각각 한 문장으로 설명할 수 있는가?
- 원본 없이 중복 없는 추첨 함수를 다시 작성할 수 있는가?
- JSON 로딩 실패의 확인 순서를 제시할 수 있는가?
- 입력 범위 변경 계획을 세우고 두 파일의 일관성을 검증할 수 있는가?
- 본인이 이해하고 수정한 부분과 AI가 생성한 부분을 구분해 설명할 수 있는가?

## 권장 학습 방법과 진도 확인

코드를 먼저 예측하고 실행한 뒤 결과가 다른 이유를 적는다. 퀴즈 답변에서 파일명, 함수명, 값의 흐름을 구체적으로 말할 수 있으면 다음 단계로 넘어간다. 마지막에는 원본을 보지 않고 핵심 추첨 함수를 다시 구현하고 기존 결과와 비교한다.

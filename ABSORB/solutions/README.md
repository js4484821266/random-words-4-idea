# 이해 확인 퀴즈 해설

문제는 [`../README.md`](../README.md)의 번호와 대응한다.

## 1. 사전 JSON 변환

예시 답안은 GitHub Pages가 서버 변환을 수행하지 않는다는 점과, 브라우저가 매번 큰 TSV의 불필요한 ID·빈도 열까지 파싱하지 않게 한다는 점을 포함해야 한다. 관련 코드는 [`../../scripts/build_words.py`](../../scripts/build_words.py)다.

흔한 오해는 JSON이면 네트워크 전송이 사라진다고 생각하는 것이다. 전송은 여전히 필요하지만 크기와 브라우저 처리량이 줄어든다.

## 2. 로딩 실행 흐름

`app.js`가 실행되면 이벤트를 등록하고 `loadWords()`를 호출한다. HTML에서 이미 비활성화된 입력과 버튼, 로딩 문구가 보인다. `fetch()`와 JSON 검사가 성공하면 배열을 저장하고 조작을 활성화하며 안내 문구를 지운다.

스스로 채점할 때 비활성 초기 상태, `fetch`, 배열 검사, 활성화와 화면 갱신을 모두 언급했는지 확인한다.

## 3. 잘못된 JSON 경로

사용자는 결과 영역의 로딩 실패 문구와 새로고침·파일 확인 안내를 본다. 개발자는 Network 응답 상태와 `app.js`의 `fetch` 경로, 실제 파일 위치와 대소문자를 먼저 비교한다.

단순히 여러 번 새로고침하는 답은 원인 확인 절차가 없으므로 충분하지 않다.

## 4. 중복 인덱스

`Set.add()`는 기존 값과 같은 인덱스를 추가하지 않는다. 크기는 그대로이며 `while` 조건이 계속 참이므로 다음 난수를 생성한다. 새 인덱스가 추가되어 목표 개수에 도달할 때 반복이 끝난다.

## 5. 입력 범위를 1~50으로 변경

[`../../index.html`](../../index.html)의 입력 `max`를 `50`으로 바꾸고, [`../../app.js`](../../app.js)의 `count > 100` 조건과 오류 문구를 `50` 기준으로 바꾼다. HTML만 바꾸면 개발자 도구나 직접 호출로 검증을 우회할 수 있고, JavaScript만 바꾸면 브라우저 기본 UI가 잘못된 범위를 안내한다.

## 6. 균등 추첨과 빈도 가중 추첨

현재 방식은 JSON의 각 단어 유형이 같은 선택 확률을 가진다. 빈도 가중 방식은 뉴스에서 많이 등장한 단어가 더 자주 선택된다. 이 프로젝트는 흔한 단어 재현보다 서로 낯선 단어 조합을 아이디어 자극으로 사용하는 것이 목적이므로 유형별 균등 추첨을 택했다.

좋은 답은 두 확률 모델의 차이, 프로젝트 목적, 빈도 열을 사용하지 않는 코드 사실을 함께 설명한다.

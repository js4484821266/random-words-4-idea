# AGENTS.md

이 문서는 AI 코딩 에이전트가 이 저장소에서 작업할 때 반드시 따라야 하는 공통 작업 지시이다.

목표는 단순히 코드를 생성하는 것이 아니라, 사용자가 이해하고 유지보수할 수 있는 형태로 안전하게 수정하는 것이다.

특히 이 저장소에서는 현재 코드만 보고 그럴듯하게 고치는 것보다, 사용자가 처음에 왜 이 프로젝트를 만들었는지, 어떤 방향을 지키려 했는지, 무엇을 망가뜨리면 안 되는지를 보존하는 것을 중요하게 본다.

## 1. 작업 범위

* repo는 이 `AGENTS.md`가 위치한 디렉터리를 최상위로 간주한다.
* 사용자 승인 없이 repo 밖 절대 경로에 파일을 만들거나 쓰지 않는다.
* 작업 전 현재 작업 디렉터리와 repo 최상위 경로를 확인한다.
* 파일 생성, 수정, 삭제 대상이 repo 내부인지 먼저 확인한다.
* C 드라이브, 사용자 홈, AppData, 시스템 임시 폴더 같은 위치를 도구나 라이브러리가 자동으로 쓰려 할 때는 repo 내부 경로로 우회하거나 사용자에게 먼저 알린다.
* 사용자가 명시하지 않은 파일명 변경, 디렉터리 구조 변경, 대규모 리팩터링은 하지 않는다.
* 사용자가 요청하지 않은 기능 추가는 하지 않는다.
* 사용자가 이미 정한 파일명, 출력 형식, 데이터 규칙, 실행 흐름은 임의로 바꾸지 않는다.

## 2. 기존 파일 수정 원칙

* 기존 파일을 수정하기 전에는 수정 대상 파일 목록과 수정 의도를 먼저 파악한다.
* 기존 코드의 목적, 입출력, 파일명 규칙, 데이터 규칙을 가능한 한 유지한다.
* 작은 수정으로 해결 가능한 문제를 큰 구조 변경으로 해결하지 않는다.
* 불필요한 스타일 변경, 포맷팅 변경, import 재정렬, 이름 변경을 피한다.
* 기존 코드가 학습용으로 작성된 경우, 지나치게 압축하거나 고급 기법으로 바꾸지 않는다.
* 사용자가 "모든 오류를 해결해서 쓰고 싶다"고 말하면 단순 skip이 아니라 원인별 복구 방법을 우선 찾는다.
* 현재 코드가 다소 투박하더라도, 사용자가 이해할 수 있는 구조라면 그 장점을 보존한다.
* 수정 전후로 무엇을 왜 바꿨는지 사용자가 따라갈 수 있어야 한다.

## 3. 파일 생성과 산출물 관리

* 테스트 산출물은 가능한 한 repo 내부의 명시된 경로에만 만든다.
* 임시 산출물을 만들면 경로와 용도를 사용자에게 알린다.
* 원본 이미지, 원본 CSV, 원본 모델 파일, 원본 데이터셋은 덮어쓰지 않는다.
* 변환 결과는 별도 output 디렉터리나 사용자가 지정한 경로에 저장한다.
* 기존 결과 파일을 덮어써야 한다면 먼저 사용자에게 확인한다.
* CSV 등 실험 결과 파일은 기존 값 규칙을 보존한다.
* 행 순서, 컬럼명, 파일명, NA 표기, 성공/실패 값 규칙을 임의로 바꾸지 않는다.
* 콘솔 요약, 성공률, 통계 요약 같은 부가 정보는 원본 CSV에 섞지 말고 출력 전용 또는 별도 파일로 둔다.

## 4. 설계 의도와 프롬프트 기록

* 사용자가 프로젝트의 최초 설계 취지, 핵심 목적, 구현 방향, 중요한 제약 조건을 설명한 경우, 이를 repo 최상위의 `./.prompts.md`에 추가한다.

* 재사용 가능한 프롬프트, 사용자 지시 프롬프트, 실험용 프롬프트도 repo 최상위의 `./.prompts.md`에 누적 기록한다.

* `./.prompts.md`가 없으면 새로 만들되, repo 밖에는 만들지 않는다.

* `./.prompts.md`는 저장소에 커밋되지 않도록 repo 최상위의 `.gitignore`에 추가한다. 이미 등록되어 있으면 중복해서 추가하지 않는다.

* `./.prompts.md`는 단순 작업 로그가 아니라, 나중에 코드를 수정할 때 최초 의도와 판단 기준을 잊지 않기 위한 설계 맥락 기록이다.

* `./.prompts.md`는 현재 코드의 모양보다 먼저 있었던 의도, 문제의식, 금지해야 할 방향, 되살려야 할 목표를 보존하는 곳이다.

* 새 내용을 추가할 때는 기존 내용을 덮어쓰거나 삭제하지 말고 파일 끝에 append한다.

* 사용자가 직접 남긴 표현이 중요하면 원문을 보존하고, 길거나 산만한 경우에는 핵심 요약을 함께 적는다.

* 각 항목에는 가능한 한 다음 정보를 포함한다.

  * 작성일시 또는 기록 일시
  * 시간대. 특별한 이유가 없으면 GMT+9 기준
  * 목적
  * 사용자 원문 또는 핵심 요약
  * 프롬프트 본문
  * 이 의도가 영향을 주는 파일, 기능, 데이터 규칙
  * 이후 수정 시 지켜야 할 기준
  * 하면 안 되는 변경 방향

* 사용자의 일시적인 감정 표현, 잡담, 확정되지 않은 아이디어는 무조건 기록하지 않는다.

* 코드 변경과 직접 관련된 설계 의도, 유지보수 기준, 금지해야 할 방향, 되살려야 할 초기 목표, 재사용 가치가 있는 프롬프트만 기록한다.

* `./.prompts.md`를 수정할 때도 repo 내부 파일 수정 원칙을 따른다.

* 사용자가 명시적으로 원하지 않는 한 `./.prompts.md`의 내용을 바탕으로 새로운 기능을 임의로 추가하지 않는다.

* `./.prompts.md`의 내용과 현재 코드가 충돌하면, 먼저 충돌 사실을 사용자에게 설명하고 어느 쪽을 우선할지 확인한다.

* 확실하지 않은 의도는 단정해서 기록하지 말고, "추정"이라고 표시하거나 사용자에게 확인한다.

````
## 5. 실행과 종료 방식

- `sys.exit()` 또는 `raise SystemExit`로 IDE 실행을 끊지 않는다.
- 가능하면 `main() -> int` 반환 방식으로 종료 상태를 표현한다.
- 학습용 또는 IDE에서 직접 실행할 가능성이 큰 스크립트의 진입점은 다음 형태를 우선 사용한다.

```python
def main() -> int:
    ...
    return 0


if __name__ == "__main__":
    main()
````

* 실제 프로세스 exit code 전달이 꼭 필요한 CLI 도구에서만 `raise SystemExit(main())` 사용 여부를 사용자에게 먼저 설명한다.
* 오류는 조용히 무시하지 않는다.
* 사용자가 원인을 파악할 수 있도록 오류 메시지를 충분히 남긴다.
* 실패를 정상 결과처럼 포장하지 않는다.

## 6. 의존성 관리

* 사용자 승인 없이 패키지를 설치하거나 업그레이드하지 않는다.
* `pip`, `conda`, `npm`, `gradle` 등 의존성 변경 명령은 실행 전 사용자에게 확인한다.
* 설치가 필요하면 패키지명, 필요한 이유, 예상 영향, 대체 방법을 먼저 설명한다.
* 기존 `requirements.txt`, `environment.yml`, `pyproject.toml`, `package.json`, `build.gradle`이 있으면 먼저 읽고 따른다.
* 패키지 버전 충돌 가능성이 있으면 최신 버전을 무작정 권하지 않는다.
* Windows + Anaconda 환경에서는 패키지 버전, 모델 파일 존재 여부, API 변경 여부를 먼저 확인한다.
* 의존성 문제와 코드 문제를 섞어서 판단하지 않는다.

## 7. 모델과 외부 파일

* 모델, 가중치, 데이터셋 파일은 자동 다운로드보다 명시적 로컬 경로를 선호한다.
* 필요한 파일명, 예상 위치, 다운로드 출처, 대체 방법을 분명히 안내한다.
* 사용자 승인 없이 네트워크 다운로드를 실행하지 않는다.
* 인터넷 연결이 없거나 차단된 환경에서도 가능한 로컬 검증 경로를 우선 고려한다.
* 외부 URL에서 받은 파일을 repo에 추가해야 할 경우, 파일 크기와 용도를 먼저 설명한다.
* 모델 파일이 없어서 기능이 제한되는 경우에는, 임시 우회 결과와 정식 실행 결과를 구분해서 설명한다.

## 8. 경로 처리

* 경로는 가능하면 `pathlib.Path`를 사용한다.
* Windows 경로 문자열을 직접 작성할 때는 raw string 또는 슬래시 변환을 사용한다.
* `/`와 `\`가 섞여도 동작하도록 작성한다.
* 한글 경로, 공백이 포함된 경로, 긴 경로를 고려한다.
* 상대 경로는 현재 작업 디렉터리에 의존하지 않도록 repo 기준으로 해석한다.
* 사용자의 터미널 화면을 직접 볼 수 있다고 가정하지 않는다.
* 직접 실행한 명령 출력만 근거로 말한다.
* 존재하지 않는 디렉터리에 파일을 저장해야 하는 경우, 디렉터리 생성 여부와 위치를 명시한다.

## 9. 코드 구조

* 학습용으로 유지보수하기 쉽게 작성한다.
* 프로젝트 산출물에 직접 영향을 주는 실행 코드는 큰 단일 파일로 만들지 않는다.
* 가능한 한 실행 코드 파일은 주석, 문서 문자열, 빈 줄을 제외한 실제 동작 코드 기준으로 파일당 100줄 이하로 나눈다.
* `.ipynb` 파일은 각 코드 셀이 주석, 문서 문자열, 빈 줄을 제외한 실제 동작 코드 기준으로 100줄 미만이 되도록 나눈다.
* 다만 지나친 분할로 흐름이 더 어려워지는 경우에는 100줄 제한보다 가독성과 응집도를 우선한다.
* 모듈 간 연결은 `import`로 명시한다.
* 각 모듈에는 핵심 의도, 입력, 출력, 실패 가능성을 설명하는 주석을 풍부하게 남긴다.
* 복잡한 조건문, 반복문, 데이터 전처리, 파일 입출력, 예외 처리에는 학습자가 흐름을 따라갈 수 있도록 주석을 충분히 넣는다.
* 주석은 코드가 무엇을 하는지만 반복하지 말고, 왜 그렇게 하는지, 어떤 입력을 예상하는지, 실패하면 무엇을 확인해야 하는지를 설명한다.
* 가능한 경우 함수 정의 시 인자와 반환값의 자료형을 명시한다.
* 단, 자료형 명시 때문에 코드가 과도하게 복잡해지거나 학습 흐름이 흐려지는 경우에는 간단한 주석 설명을 우선한다.
* 함수는 가능한 한 한 가지 책임만 갖도록 작성한다.
* 전역 상태와 숨은 부작용을 줄인다.
* 하드코딩된 절대 경로를 피한다.
* 성능 최적화보다 먼저 정확성, 재현성, 이해 가능성을 우선한다.

## 10. ABSORB 강의교재 작성 규칙

* repo 최상위의 `ABSORB/` 디렉터리에 사용자가 이 프로젝트의 코드를 실제로 이해하고 자기 지식으로 흡수할 수 있도록 강의교재를 작성한다.

* 기존의 단일 `ABSORB.md` 대신 `ABSORB/README.md`를 교재의 시작 문서로 사용한다.

* `ABSORB/README.md`는 현재 프로젝트를 학습하기 위한 입구이자, `ABSORB/` 안의 모든 학습 문서를 순서대로 안내하는 목차이다.

* 교재의 목적은 프로젝트를 완성품처럼 홍보하거나 결과만 해설하는 것이 아니다. 사용자가 코드의 구조, 실행 흐름, 핵심 개념, 구현 이유를 이해하고, 나중에는 도움 없이 설명하고 수정하고 일부를 다시 구현할 수 있게 만드는 것이다.

* 사용자가 이 프로젝트를 자신의 실력이라고 말할 수 있으려면, 최소한 다음을 할 수 있어야 한다.

  * 프로젝트가 해결하는 문제를 자신의 말로 설명하기
  * 전체 구조와 실행 흐름을 설명하기
  * 주요 파일, 모듈, 함수, 클래스의 역할을 설명하기
  * 핵심 구현을 왜 그렇게 작성했는지 설명하기
  * 입력이 어떤 과정을 거쳐 출력으로 바뀌는지 추적하기
  * 오류가 발생했을 때 원인을 좁혀 가기
  * 작은 기능을 직접 수정하거나 추가하기
  * 핵심 부분을 빈 파일에서 다시 구현하기
  * 면접에서 코드와 기술적 선택을 과장 없이 설명하기

* `./.prompts.md`가 최초 의도와 판단 기준을 보존하는 문서라면, `ABSORB/`는 현재 코드에 포함된 지식과 구현 방식을 사용자가 자기 것으로 만들기 위한 학습교재이다.

* `ABSORB/README.md`에서 시작하여 사용자가 링크를 순서대로 따라가면 프로젝트 전체를 단계적으로 학습할 수 있어야 한다.

* 각 학습 문서는 선행 문서에서 배운 내용을 전제로 다음 내용을 설명한다. 아직 설명하지 않은 개념을 이미 안다고 가정하거나, 코드만 제시하고 설명을 생략하지 않는다.

* 프로젝트가 사용하는 언어, 프레임워크, 라이브러리와 개념 중 사용자가 이해해야 하는 내용은 프로젝트 코드와 연결하여 설명한다. 프로젝트와 무관한 일반 이론을 과도하게 늘어놓지 않는다.

* 학습 문서 파일명에는 읽는 순서를 알 수 있도록 번호를 붙인다. 다음과 같은 구성을 우선하되, 실제 프로젝트에 맞게 조정한다.


  * 교재의 목적
  * 최종적으로 할 수 있어야 하는 것
  * 학습 대상과 필요한 선수 지식

  * 학습에 필요한 실행 환경
  * 원본 코드와 주요 산출물로 이동하는 링크
  * 권장 학습 방법

* 전체 교재에는 프로젝트 특성에 맞게 최소한 다음 내용을 포함한다.

  * 프로젝트가 해결하려는 문제와 설계 목표
  * 최초 설계 취지와 현재 코드의 관계
  * 전체 디렉터리 및 파일 구조
  * 프로그램의 시작점과 전체 실행 흐름
  * 주요 모듈, 함수, 클래스, 데이터 구조의 역할
  * 입력, 상태 변화, 처리, 저장, 출력의 흐름
  * 외부 모델, API, 라이브러리와 의존성
  * 핵심 알고리즘과 구현 선택의 이유
  * 대안으로 가능한 방식과 현재 방식의 장단점
  * 주요 예외 처리와 실패 가능성
  * 테스트 전략과 검증 방법
  * 디버깅할 때 확인할 순서
  * 작은 수정 실습
  * 일부 기능의 빈칸 구현 실습
  * 핵심 부분의 재구현 실습
  * 사용자가 자기 말로 설명하는 연습
  * 면접에서 나올 수 있는 질문과 과장 없는 답변 기준

* 문서는 소스 코드를 단순히 나열하거나 줄마다 번역하지 않는다. 코드가 왜 존재하는지, 앞뒤 코드와 어떻게 연결되는지, 입력과 출력이 무엇인지, 수정하면 어떤 영향이 생기는지를 설명한다.

* 중요한 코드는 다음 순서로 설명한다.

  * 이 코드가 필요한 이유
  * 이 코드가 받는 입력
  * 내부에서 일어나는 처리
  * 반환값 또는 부작용
  * 다른 파일이나 함수와의 연결
  * 실패할 수 있는 조건
  * 직접 바꿔 볼 수 있는 부분
  * 이해했는지 확인할 질문

* 설명만 읽고 넘어가게 하지 말고, 사용자가 직접 코드를 관찰하고 실행하고 수정하도록 구성한다.

* 각 문서에는 가능한 한 다음 요소를 포함한다.

  * 이번 문서의 학습 목표
  * 앞 문서와의 연결
  * 먼저 생각해 볼 질문
  * 핵심 개념 설명
  * 실제 프로젝트 파일과 코드에 대한 상대 링크
  * 실행하거나 관찰할 예시
  * 사용자가 직접 할 짧은 실습
  * 예상 결과와 확인 방법
  * 자주 헷갈리는 부분
  * 이해 확인 퀴즈
  * 이번 문서의 핵심 요약
  * 다음 문서로 이동하는 링크

* 각 핵심 문서에는 학습 내용을 실제로 이해했는지 확인할 수 있는 퀴즈를 포함한다.

* 퀴즈는 단순 암기 확인에 그치지 않고, 다음 유형을 프로젝트 특성에 맞게 섞어서 구성한다.

  * 개념 설명형: 용어나 구조를 자기 말로 설명한다.
  * 코드 추적형: 주어진 입력이 어떤 경로를 거쳐 어떤 출력이 되는지 예측한다.
  * 결과 예측형: 조건, 값, 호출 순서를 바꿨을 때 결과를 예상한다.
  * 오류 찾기형: 잘못된 코드나 설정에서 원인과 수정 방향을 찾는다.
  * 연결 관계형: 파일, 함수, 클래스, 데이터 사이의 관계를 설명한다.
  * 빈칸 구현형: 핵심 코드 일부를 직접 완성한다.
  * 수정 설계형: 작은 요구사항을 반영하려면 어디를 어떻게 바꿔야 하는지 제안한다.
  * 재구현형: 원본을 보지 않고 핵심 기능을 다시 작성한다.
  * 구두 설명형: 면접을 가정해 기술적 선택과 장단점을 설명한다.

* 각 퀴즈에는 무엇을 확인하려는 문제인지 명확한 학습 목표를 연결한다.

* 객관식만으로 이해도를 판단하지 않는다. 핵심 개념에는 서술형, 코드 작성형, 디버깅형 문제를 반드시 포함한다.

* 정답을 우연히 맞히는 것을 줄이기 위해 답뿐 아니라 이유, 실행 흐름, 수정 근거를 함께 설명하도록 요구한다.

* 퀴즈 난이도는 가능한 한 다음과 같이 단계화한다.

  * 기본: 바로 앞에서 배운 개념을 확인한다.
  * 적용: 실제 프로젝트 코드에 개념을 적용한다.
  * 변형: 입력이나 조건이 달라진 상황을 해결한다.
  * 독립 수행: 원본 설명 없이 직접 분석하거나 구현한다.

* 각 장의 마지막에는 짧은 장별 퀴즈를 두고, 주요 학습 단위가 끝나는 지점에는 여러 문서의 내용을 연결하는 종합 퀴즈를 둔다.

* 전체 교재 마지막에는 프로젝트 전체를 대상으로 하는 최종 점검을 포함한다. 최종 점검은 최소한 다음을 확인한다.

  * 프로젝트 목적을 자기 말로 설명할 수 있는가
  * 전체 실행 흐름을 코드 위치와 함께 설명할 수 있는가
  * 핵심 구현을 원본 없이 일부 재현할 수 있는가
  * 오류 상황에서 확인 순서를 제시할 수 있는가
  * 작은 기능 변경 계획을 세우고 구현할 수 있는가
  * 주요 기술 선택의 이유와 대안을 설명할 수 있는가
  * 자신의 기여와 AI 또는 외부 도움의 범위를 정직하게 구분할 수 있는가

* 퀴즈 바로 아래에는 정답을 노출하지 않는다. 정답과 해설은 기본적으로 `ABSORB/solutions/` 아래에 문서별로 분리한다.

* 해설 파일은 원래 문제 번호와 일치시켜 사용자가 쉽게 대조할 수 있게 작성한다.

* 해설에는 정답만 적지 않고 다음을 포함한다.

  * 정답 또는 예시 답안
  * 그렇게 판단하는 이유
  * 관련 코드 위치
  * 흔히 하는 오답과 오해
  * 다른 가능한 해결 방식
  * 스스로 채점할 수 있는 기준

* 서술형이나 구현형 문제처럼 답이 하나가 아닌 경우에는 단일 정답 대신 필수 포함 요소와 평가 기준을 제공한다.

* 필요하면 `ABSORB/quiz/`를 별도 디렉터리로 만들어 장별 퀴즈와 종합 퀴즈를 모을 수 있다. 다만 각 본문 문서에서도 해당 퀴즈로 바로 이동할 수 있어야 한다.

* 실습은 단순 복사와 실행에 그치지 않도록 단계적으로 구성한다.

  * 1단계: 코드를 읽고 실행 결과를 예측한다.
  * 2단계: 작은 값이나 조건을 바꾸고 결과 차이를 관찰한다.
  * 3단계: 일부 코드를 빈칸으로 두고 직접 완성한다.
  * 4단계: 기존 구현을 보지 않고 핵심 기능을 다시 작성한다.
  * 5단계: 구현 선택과 결과를 자신의 말로 설명한다.

* 모든 문서에 억지로 많은 문제를 넣을 필요는 없지만, 핵심 기능마다 최소한 하나 이상의 능동적 실습과 하나 이상의 이해 확인 퀴즈를 제공한다.

* 정답을 바로 노출하면 학습 효과가 떨어지는 실습은 문제와 해설을 분리한다. 필요한 경우 `ABSORB/solutions/`를 만들고 해당 문서에서 해설 링크를 제공한다.

* 사용자가 구현을 외우는 데 그치지 않도록, 변수명이나 입력 형식을 바꾼 변형 문제와 작은 확장 과제를 포함한다.

* 핵심 구현은 가능한 경우 다음 세 수준으로 다룬다.

  * 읽을 수 있다.
  * 설명할 수 있다.
  * 다시 구현할 수 있다.

* 교재가 사용자의 실제 이해를 보장한다고 단정하지 않는다. 대신 사용자가 이해 여부를 검증할 수 있는 질문, 실습, 재구현 과제와 확인 기준을 제공한다.

* 가능하면 구조 다이어그램, 실행 흐름도, 데이터 흐름도와 표를 포함한다.

* 각 코드 파일, 미디어 리소스, 데이터 파일, 설정 파일, 주요 식별자에 대한 상대 링크를 포함한다.

* 문서 안의 링크는 `ABSORB/` 디렉터리를 기준으로 실제 repo 파일에 접근할 수 있도록 올바른 상대 경로로 작성한다.

* 학습자가 코드를 직접 열어보지 않아도 전체 구조를 파악할 수 있게 설명하되, 중요한 설명에는 반드시 실제 코드 위치를 연결한다.

* 처음 등장하는 용어는 바로 정의한다. 앞에서 배운 용어는 필요한 경우 짧게 복습하고 다음 개념과 연결한다.

* 프로젝트 코드가 바뀌어 교재 내용, 링크, 실습 또는 해설과 불일치하게 된 경우 관련 `ABSORB/` 문서도 함께 갱신한다.

* 단순한 한 파일짜리 수정처럼 교재 갱신이 과한 경우에는 갱신하지 않아도 되지만, 그 이유를 작업 완료 보고의 `변경 이유` 또는 `남은 문제`에 적는다.

* 기존 repo 최상위에 `ABSORB.md`가 있으면 내용을 잃지 않도록 검토한 뒤 `ABSORB/README.md`와 관련 학습 문서로 이전한다. 이전이 끝났다고 확인하기 전에는 기존 파일을 삭제하지 않는다.
## 11. 테스트와 검증

* 변경 후에는 가능한 가장 작은 단위의 테스트부터 실행한다.
* 전체 테스트가 오래 걸리거나 외부 리소스를 요구하면, 먼저 빠른 검증 방법을 제안한다.
* 테스트 명령을 실행했다면 사용한 명령과 핵심 결과를 요약한다.
* 테스트가 불가능하면 그 이유와 수동 확인 방법을 적는다.
* 실행하지 못한 테스트나 확인하지 못한 결과를 성공했다고 말하지 않는다.
* 같은 오류에 대해 무작정 여러 번 재시도하지 않는다.
* 오류가 나면 환경 문제, 경로 문제, 버전 문제, 입력 파일 문제, 코드 문제를 분리해서 본다.
* 테스트 결과가 부분 성공이면 부분 성공이라고 말한다.
* 샘플 입력으로만 확인한 경우, 전체 데이터에서도 성공한다고 단정하지 않는다.

## 12. 오류 처리와 보고

* 오류 메시지 원문을 가능한 한 보존한다.
* 추정 원인과 확인된 사실을 구분한다.
* 해결한 오류와 아직 남은 오류를 구분한다.
* 임시로 우회한 부분이 있으면 반드시 알린다.
* 단순히 "수정했습니다"라고 하지 말고 무엇을 왜 바꿨는지 설명한다.
* 모르는 것은 추측으로 확정하지 않는다.
* 사용자가 원인을 재현하거나 확인할 수 있도록 파일명, 명령, 입력 조건을 함께 남긴다.

## 13. 스타일

* 가능한 한 lint 실행 시 불필요한 경고가 나오지 않도록 깔끔한 코드 스타일을 유지한다.
* 포맷터가 이미 정해져 있으면 기존 설정을 따른다.
* 새 포맷터나 린터를 도입하지 않는다.
* 변수명은 의미가 드러나게 짓는다.
* 지나치게 짧은 약어를 피한다.
* 학습자가 흐름을 따라갈 수 있도록 명확하고 단순한 코드를 우선한다.
* 스타일 통일을 이유로 사용자가 요청하지 않은 대규모 변경을 하지 않는다.

## 14. 작업 완료 보고 형식

작업 완료 후에는 다음 형식으로 보고한다.

````md
## 변경한 파일

- `path/to/file.py`
  - 변경 내용 요약

## 변경 이유

- 왜 이 변경이 필요했는지 설명

## 실행한 검증

```bash
검증 명령
```

## 검증 결과

- 성공 또는 실패 여부
- 핵심 출력 요약

## 남은 문제

- 아직 해결하지 못한 문제
- 사용자가 직접 확인해야 할 점
- 추가로 필요한 파일이나 설정
````

검증을 실행하지 못했다면 `실행한 검증` 항목에 실행하지 못한 이유를 적는다.

## 15. 금지 사항

* 사용자 승인 없는 repo 밖 파일 생성
* 사용자 승인 없는 패키지 설치 또는 업그레이드
* 사용자 승인 없는 네트워크 다운로드
* 원본 데이터 덮어쓰기
* 실험 CSV 규칙 임의 변경
* 확인하지 않은 결과를 성공했다고 말하기
* 오류를 숨기고 skip 처리하기
* 사용자가 요청하지 않은 대규모 리팩터링
* 사용자가 요청하지 않은 기능 추가
* 터미널 화면을 직접 본 것처럼 말하기
* IDE 실행을 강제로 끊는 종료 방식 사용
* `./.prompts.md`의 설계 의도를 무시하고 현재 코드만 기준으로 방향을 바꾸기
* `ABSORB/` 강의교재와 `./.prompts.md`를 같은 역할의 문서처럼 섞어 쓰기
